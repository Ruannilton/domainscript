// rabbitmq.go.txt implementa runtime.ChannelTransport sobre uma conexão
// RabbitMQ real (Marco J, tasks J3.1/J3.2, REQ-43.1/43.3/43.4, §design
// infra-providers 3.3) — o contraparte cross-process de
// runtime.NewQueueChannel (rtsrc/channel.go.txt, que só entrega dentro do
// MESMO processo, limitação documentada explicitamente ali). Selecionado
// pelo wiring gerado (task J3.4, codegen/channel.go) quando um canal da
// topologia declara `via: queue provider: "rabbitmq"`; sem esse provider, o
// QueueChannel in-memory de sempre continua (NFR-21).
//
// --- Ordenação por partição × concorrência (task J3.2, REQ-43.3, R6) ---
//
// Mesmo dilema que a doc de queueChannel (rtsrc/channel.go.txt) já resolve
// in-memory: um consumidor ativo dá ordem mas mata a concorrência; N
// consumidores dão concorrência mas embaralham a ordem por chave. A regra
// (idêntica à do in-memory, agora cross-process):
//   - cfg.KeyFunc != nil (`orderBy` declarado na topologia) ⇒ a exchange é
//     declarada "x-consistent-hash" (plugin oficial do RabbitMQ) em vez de
//     "fanout": Publish roteia com routing key = KeyFunc(ev), e a exchange
//     hasheia essa chave para escolher DETERMINISTICAMENTE uma entre
//     cfg.Concurrency filas de partição (partitionQueueNames) — cada
//     partição tem exatamente UM consumidor (goroutine + *amqp.Channel
//     próprios, Qos prefetch=1), preservando ordem DENTRO da partição.
//     Publicações para a MESMA chave sempre hasheiam para a MESMA partição
//     (propriedade da exchange consistent-hash), então a ordem de Publish
//     é preservada na entrega — o mesmo contrato que queueChannel.shardFor
//     já garante in-memory.
//   - cfg.KeyFunc == nil (sem `orderBy`) ⇒ exchange "fanout" simples, UMA
//     fila (cfg.Queue), mas cfg.Concurrency *amqp.Channel/consumidores
//     DISTINTOS todos consumindo essa MESMA fila (RabbitMQ distribui round-
//     robin entre consumidores concorrentes de uma fila — "competing
//     consumers") — atinge o mesmo teto de paralelismo que "prefetch =
//     workers.concurrency" (§design 3.3) pediria de um único consumidor com
//     QoS N + pool de goroutines, só que sem precisar sincronizar
//     Ack/Nack/Publish entre goroutines que dividiriam o MESMO *amqp.Channel
//     (cada consumidor tem o seu, dono exclusivo de uma única goroutine —
//     nenhum mutex extra necessário além de pubMu, que já protege só o
//     canal de Publish).
//
// --- Poison pill / DLX + retry-queue(TTL) + DLQ final (task J3.2, REQ-43.4) ---
//
// Substitui o nack(requeue=false)-descarta-direto de J3.1: TODA fila
// principal (partição ou não) é declarada com `x-dead-letter-exchange`
// apontando para uma DLX própria do canal (retryExchangeName) — um
// nack(requeue=false) não descarta mais, ele roteia para essa DLX, que
// alimenta UMA retry queue (retryQueueName) com `x-message-ttl` +
// `x-dead-letter-exchange` apontando de VOLTA para a exchange original: ao
// expirar o TTL (sem nenhum consumidor na retry queue — ela só existe para
// segurar a mensagem pelo tempo do TTL), o broker reencaminha a mensagem
// para a exchange original, incrementando `x-death` a cada passagem por
// esse ciclo — e, como nem a fila principal nem a retry queue sobrescrevem
// a routing key original (`x-dead-letter-routing-key` nunca é setado), uma
// mensagem particionada por consistent-hash volta para a MESMA partição
// depois de reencaminhada (a chave original é preservada pelo mecanismo de
// dead-lettering do próprio broker). xDeathCount lê esse header para saber
// quantas vezes a mensagem já deu a volta; ao atingir
// effectiveMaxAttempts(cfg.MaxAttempts) (REQ-43.4 reusa o CAMPO
// `circuitBreaker.threshold` do canal como contador de tentativas — não a
// máquina de estados open/half-open/closed de runtime.CircuitBreaker, que
// resolve um problema diferente: um breaker por-canal global guardando
// TODA entrega, não um contador por-mensagem), o consumidor publica o
// corpo DIRETO na DLQ final (dlqName, via exchange default "" — routing
// direto por nome de fila) e dá ack na mensagem original, removendo-a
// definitivamente do ciclo de retry.
//
// --- Envelope (task J3.1.b, §design 3.3) ---
//
// JSON `{eventType, payload}` — o mesmo shape que o Dispatcher já move em
// memória, serializado (todo runtime.Event já é JSON-serializável, convenção
// de E4.2: EventMeta + os campos do próprio tipo). encodeEnvelope/
// decodeEnvelope, abaixo, são funções livres (não métodos) de propósito: o
// teste da task J3.1.e as exercita isoladamente, round-trip, sem abrir
// nenhuma conexão AMQP real — mesmo espírito para as funções puras de
// montagem de topologia desta task (partitionQueueNames/mainQueueArgs/
// retryQueueArgs/xDeathCount/consistentHashBindingKey), exercitadas pelo
// teste da task J3.2.c sem NENHUMA conexão real.
//
// --- Registry consumidor: contracts.EventRegistry() + o do módulo (R8) ---
//
// decodeEnvelope recebe o registry pronto — quem o monta é o wiring gerado
// (task J3.4, decl_policy.go/generateCmdMainFile): REQ-43.5 exige que o
// binário CONSUMIDOR (um processo diferente do produtor) registre as
// factories de contracts/ (os PublicEvent) ALÉM do EventRegistry() do
// próprio módulo — um evento cross-service que só existe como PublicEvent
// nunca teria sua factory no EventRegistry() do módulo consumidor sozinho
// (EventRegistry() só cobre Event/PublicEvent DECLARADOS por aquele módulo,
// mesmo raciocínio já documentado em decl_policy.go sobre por que o outbox
// durável monta seu registry inline em vez de reusar EventRegistry()).
//
// --- Reconexão (task J3.3, REQ-43.6, §design 3.3) ---
//
// O `amqp091-go` não reconecta sozinho: um blip de rede fecha `*amqp.
// Connection`/`*amqp.Channel` em silêncio (o consumidor simplesmente pararia
// de receber, sem nenhum erro visível) — supervise (abaixo) resolve isso
// registrando `Connection.NotifyClose` na conexão ATUAL e, ao detectar um
// fechamento que NÃO veio de Close() (r.stopCh fecha ANTES de r.conn.Close()
// dentro de Close(), então checar r.stopCh logo após o disparo de
// NotifyClose distingue as duas causas de forma confiável), entra em
// reconnectLoop: redial + re-declaração da topologia INTEIRA (mesmo código
// de declareTopology que NewRabbitMQChannel já usa — nenhuma duplicação) +
// reabertura de todos os consumidores, com backoff exponencial (mesmo
// idioma de DurableOutbox.Start, rtsrc/outbox.go.txt: dobra a cada falha,
// capado em reconnectMaxBackoff, nunca martelando o broker numa queda
// prolongada). Durante a janela de reconexão r.conn/r.ch ficam nil sob
// connMu — Publish lê isso e retorna erro IMEDIATAMENTE (nunca bloqueia,
// nunca panica sobre um *amqp.Channel nil): é o comportamento que REQ-43.6
// pede, e é suficiente porque o único chamador que importa (o relay de
// DurableOutbox.Tick, rtsrc/outbox.go.txt) já re-tenta uma linha não
// entregue na próxima Tick — nenhum buffer/fila própria de Publish é
// necessário aqui. Subscribe não é afetado (só popula um map protegido por
// mu, nunca toca conn/ch). connMu (RWMutex) é o mutex NOVO desta task —
// pubMu continua existindo, mas passa a serializar só a chamada de I/O em si
// (PublishWithContext), enquanto connMu serializa as TROCAS de conn/ch/
// consumeChans entre o supervisor e Publish/Close (conn/ch deixaram de ser
// write-once na construção a partir desta task).
//
// Testável sem broker: nextReconnectBackoff é uma função livre (a sequência
// de espera do loop, mesmo espírito das outras funções puras desta doc) —
// exercitada por TestNextReconnectBackoffDoublesAndCaps
// (amqp_reconnect_test.go). O resto (supervise/reconnectLoop de verdade)
// depende de um socket de verdade fechando: `*amqp.Connection`/`*amqp.
// Channel` são STRUCTS concretos do driver (não interfaces), então não há
// como fabricar um NotifyClose "fake" sem abrir uma conexão real — esse
// teste de integração fica fora do orçamento desta task (J3.4/J6).

// EventFactory espelha runtime.EventFactory/sqlruntime.EventFactory: constrói
// um Event zerado (ponteiro) para um tipo de evento — o que decodeEnvelope
// usa para materializar o payload JSON no tipo Go concreto certo.
type EventFactory = func() runtime.Event

// envelope é o formato de fio (task J3.1.b, §design 3.3): eventType nomeia o
// tipo (EventType(), a chave de registry) e payload é o próprio Event
// serializado — RawMessage para não decodificar duas vezes (uma vez pra
// achar eventType, outra pro tipo concreto).
type envelope struct {
	EventType string          `json:"eventType"`
	Payload   json.RawMessage `json:"payload"`
}

// encodeEnvelope serializa ev no envelope de fio (task J3.1.b).
func encodeEnvelope(ev runtime.Event) ([]byte, error) {
	if ev == nil {
		return nil, fmt.Errorf("amqpruntime: encodeEnvelope: ev nil")
	}
	payload, err := json.Marshal(ev)
	if err != nil {
		return nil, fmt.Errorf("amqpruntime: encodeEnvelope: %w", err)
	}
	return json.Marshal(envelope{EventType: ev.EventType(), Payload: payload})
}

// decodeEnvelope reconstrói o runtime.Event concreto que data carrega, via
// registry (ver a doc do arquivo sobre R8/REQ-43.5 — este código não monta
// registry, só consome o que recebe). Um eventType desconhecido é uma
// condição PERMANENTE — nada num decodeEnvelope futuro decodificaria essa
// linha de forma diferente — mesma classificação de "poison pill" que
// DurableOutbox.deliver já documenta (rtsrc/outbox.go.txt, Marco J); assim
// como uma falha de handler, essa condição passa pelo MESMO ciclo de
// DLX+retry+DLQ final de consume (abaixo) — nunca um caminho especial só
// para falha de decodificação, mais simples e ainda limitado (nunca gira
// para sempre: effectiveMaxAttempts sempre a manda pra DLQ final depois de
// N tentativas, decodificação nunca teria sucesso numa retentativa, mas o
// custo de tentar de novo é só round-trips de TTL, não um loop apertado).
func decodeEnvelope(data []byte, registry map[string]EventFactory) (runtime.Event, error) {
	var env envelope
	if err := json.Unmarshal(data, &env); err != nil {
		return nil, fmt.Errorf("amqpruntime: decodeEnvelope: envelope malformado: %w", err)
	}
	factory, ok := registry[env.EventType]
	if !ok || factory == nil {
		return nil, fmt.Errorf("amqpruntime: decodeEnvelope: tipo de evento desconhecido %q (registry incompleto? ver a doc do arquivo sobre contracts.EventRegistry(), REQ-43.5)", env.EventType)
	}
	ev := factory()
	if ev == nil {
		return nil, fmt.Errorf("amqpruntime: decodeEnvelope: factory de %q devolveu nil (registry mal configurado)", env.EventType)
	}
	if err := json.Unmarshal(env.Payload, ev); err != nil {
		return nil, fmt.Errorf("amqpruntime: decodeEnvelope: payload de %q malformado: %w", env.EventType, err)
	}
	return ev, nil
}

// --- Montagem de topologia (task J3.2.c): funções puras, testáveis sem
// nenhuma conexão AMQP real — cada uma só monta um nome ou um amqp.Table de
// argumentos, nunca chama um método de rede. ---

const (
	// defaultRetryTTL é o TTL da retry queue quando cfg.RetryTTL <= 0 —
	// nenhum canal .ds controla isto ainda (a espera é que a task J3.4
	// eventualmente leia um valor de `circuitBreaker`/topology.ds; até lá,
	// este default cobre o caminho "declarado sem essa config").
	defaultRetryTTL = 5 * time.Second
	// defaultMaxAttempts é o teto de tentativas quando cfg.MaxAttempts <= 0
	// — mesma razão de defaultRetryTTL.
	defaultMaxAttempts int64 = 5
	// consistentHashWeight é o peso de binding usado em TODA fila de
	// partição (uniforme — cada partição recebe a mesma fração esperada do
	// hash space; a exchange x-consistent-hash exige que a binding key seja
	// um número textual representando o peso, não um padrão de rota).
	consistentHashWeight = "10"
	// reconnectInitialBackoff/reconnectMaxBackoff (task J3.3, REQ-43.6):
	// mesmo idioma de defaultOutboxPollInterval/defaultOutboxMaxBackoff
	// (rtsrc/outbox.go.txt) — reconnectLoop dobra o intervalo a cada
	// tentativa de redial que falha, começando em reconnectInitialBackoff,
	// nunca ultrapassando reconnectMaxBackoff, para não martelar um broker
	// fora do ar.
	reconnectInitialBackoff = time.Second
	reconnectMaxBackoff     = 30 * time.Second
)

// effectiveRetryTTL normaliza cfg.RetryTTL (<=0 vira defaultRetryTTL).
func effectiveRetryTTL(cfg time.Duration) time.Duration {
	if cfg <= 0 {
		return defaultRetryTTL
	}
	return cfg
}

// effectiveMaxAttempts normaliza cfg.MaxAttempts (<1 vira
// defaultMaxAttempts) — mesma convenção de clamping que
// runtime.NewCircuitBreaker já usa para threshold < 1.
func effectiveMaxAttempts(cfg int64) int64 {
	if cfg < 1 {
		return defaultMaxAttempts
	}
	return cfg
}

// effectiveConcurrency normaliza cfg.Concurrency (<1 vira 1) — mesma
// convenção de QueueChannelConfig.Concurrency (rtsrc/channel.go.txt).
func effectiveConcurrency(cfg int64) int64 {
	if cfg < 1 {
		return 1
	}
	return cfg
}

// partitionQueueNames devolve as N filas de partição de base (ordem
// estável, "<base>-p0".."<base>-p(n-1)") — usadas quando o canal declara
// `orderBy` (cfg.KeyFunc != nil).
func partitionQueueNames(base string, n int) []string {
	names := make([]string, n)
	for i := range names {
		names[i] = fmt.Sprintf("%s-p%d", base, i)
	}
	return names
}

// retryExchangeName/retryQueueName/dlqName derivam, deterministicamente, os
// três recursos extras de DLX+retry+DLQ final a partir do nome base da
// exchange do canal — nenhuma config adicional do canal precisa nomeá-los.
func retryExchangeName(exchangeBase string) string { return exchangeBase + "-retry-dlx" }
func retryQueueName(exchangeBase string) string     { return exchangeBase + "-retry" }
func dlqName(exchangeBase string) string             { return exchangeBase + "-dlq" }

// mainQueueArgs monta os argumentos de declaração de UMA fila principal
// (partição ou única): só `x-dead-letter-exchange`, apontando pra DLX de
// retry — um nack(requeue=false) nessa fila passa a rotear pra lá em vez de
// descartar (REQ-43.4).
func mainQueueArgs(retryExchange string) amqp.Table {
	return amqp.Table{"x-dead-letter-exchange": retryExchange}
}

// retryQueueArgs monta os argumentos da retry queue: `x-message-ttl` (em
// milissegundos, a unidade que o protocolo AMQP exige) + `x-dead-letter-
// exchange` apontando de VOLTA pra exchange ORIGINAL do canal — ao expirar
// o TTL, o broker reencaminha a mensagem pra lá (nunca pra DLX de retry de
// novo: só a fila principal aponta pra DLX de retry, a retry queue aponta
// pra exchange original), incrementando `x-death` a cada passagem.
func retryQueueArgs(mainExchange string, ttl time.Duration) amqp.Table {
	return amqp.Table{
		"x-dead-letter-exchange": mainExchange,
		"x-message-ttl":          ttl.Milliseconds(),
	}
}

// consistentHashBindingKey devolve a binding key (peso) usada por TODA fila
// de partição — ver a doc de consistentHashWeight.
func consistentHashBindingKey() string { return consistentHashWeight }

// nextReconnectBackoff dobra current, capado em reconnectMaxBackoff — a
// função PURA que decide a sequência de espera de reconnectLoop (task J3.3,
// REQ-43.6), extraída de propósito para ser testável sem nenhum socket real
// (ver a doc do arquivo sobre "Reconexão" — reconnectLoop só CHAMA esta
// função dentro do loop de fato, nunca reimplementa o dobramento inline).
func nextReconnectBackoff(current time.Duration) time.Duration {
	next := current * 2
	if next > reconnectMaxBackoff || next <= 0 {
		return reconnectMaxBackoff
	}
	return next
}

// xDeathCount lê o header `x-death` (formato padrão do RabbitMQ: um array
// de amqp.Table) e soma quantas vezes esta mensagem já deu a volta pelo
// ciclo DLX+retry — 0 quando o header está ausente (mensagem na 1ª
// tentativa) ou malformado (nunca panica sobre um header inesperado, só
// trata como "nenhuma tentativa anterior conhecida").
//
// Filtra por reason == "rejected" (revisão da PR #24): CADA volta do ciclo
// produz DOIS eventos de dead-lettering, não um — (1) da fila principal pra
// DLX de retry, reason "rejected" (o nack(requeue=false) de consume), e (2)
// da retry queue de volta pra exchange original, reason "expired" (o TTL
// vencendo). Somar os dois SEM filtrar contaria cada tentativa em dobro —
// uma mensagem chegaria na DLQ final depois de só metade de
// effectiveMaxAttempts tentativas de verdade. "rejected" conta exatamente
// uma vez por tentativa que de fato falhou (decodificação ou handler);
// "expired" é só o mecanismo de trânsito de volta, não uma tentativa em si.
func xDeathCount(headers amqp.Table) int64 {
	if headers == nil {
		return 0
	}
	raw, ok := headers["x-death"]
	if !ok {
		return 0
	}
	entries, ok := raw.([]interface{})
	if !ok {
		return 0
	}
	var total int64
	for _, e := range entries {
		t, ok := e.(amqp.Table)
		if !ok {
			continue
		}
		if reason, _ := t["reason"].(string); reason != "rejected" {
			continue
		}
		switch c := t["count"].(type) {
		case int64:
			total += c
		case int32:
			total += int64(c)
		case int:
			total += int64(c)
		}
	}
	return total
}

// RabbitMQConfig configura NewRabbitMQChannel a partir de um canal `via:
// queue provider: "rabbitmq"` da topologia (task J3.4 monta isto —
// Exchange/Queue nomeados a partir do canal, ex. "<from>-<to>").
type RabbitMQConfig struct {
	Exchange string
	Queue    string
	// KeyFunc espelha runtime.KeyFunc (rtsrc/channel.go.txt) — nil quando o
	// canal não declara `orderBy` (work-queue simples, sem ordem); não-nil
	// ativa o particionamento consistent-hash (ver a doc do arquivo sobre
	// ordenação por partição, REQ-43.3).
	KeyFunc runtime.KeyFunc
	// Concurrency é o nº de filas de partição (KeyFunc != nil) ou de
	// consumidores concorrentes da MESMA fila (KeyFunc == nil) — de
	// `workers.concurrency` (spec §11). < 1 vira 1 (effectiveConcurrency).
	Concurrency int64
	// MaxAttempts é o teto de tentativas antes de uma mensagem ir para a
	// DLQ final — de `circuitBreaker.threshold` (spec §11, reusado como
	// contador de tentativas, REQ-43.4, ver a doc do arquivo). <= 0 vira
	// defaultMaxAttempts.
	MaxAttempts int64
	// RetryTTL é quanto tempo uma mensagem fica retida na retry queue antes
	// de ser reencaminhada pra tentativa seguinte. <= 0 vira defaultRetryTTL.
	RetryTTL time.Duration
}

// rabbitmqChannel implementa runtime.ChannelTransport (task J3.1.a/J3.2/J3.3).
type rabbitmqChannel struct {
	// url é guardado (a partir da task J3.3) porque reconnectLoop precisa
	// redial do zero — antes desta task, NewRabbitMQChannel só usava url
	// localmente e nunca mais precisava dele depois de construído.
	url string
	cfg RabbitMQConfig
	// dlq é o nome da DLQ final (dlqName(cfg.Exchange)), guardado para
	// consume não precisar recalculá-lo a cada mensagem — não muda entre
	// reconexões (deriva só de cfg, imutável após a construção).
	dlq      string
	registry map[string]EventFactory

	// connMu (task J3.3) protege conn/ch/consumeChans: write-once na
	// construção até esta task, agora TROCADOS por reconnectLoop a cada
	// reconexão, então toda leitura (Publish, Close) e escrita (dialAndSetup
	// callers) passa por aqui. RWMutex porque Publish (o caminho quente) só
	// PRECISA ler r.ch — RLock deixa publicações concorrentes não se
	// bloquearem umas às outras por causa desta troca, só por pubMu
	// (abaixo), que é quem de fato precisa ser exclusivo.
	connMu sync.RWMutex
	conn   *amqp.Connection
	// ch é o *amqp.Channel dedicado a Publish — nunca compartilhado com
	// nenhum consumeChans (cada consumidor tem o seu, ver a doc do arquivo
	// sobre por que isso evita qualquer mutex de Ack/Nack). nil enquanto
	// reconnectLoop está no ar (task J3.3) — Publish trata isso como "sem
	// conexão", devolvendo erro em vez de bloquear ou desreferenciar nil.
	ch *amqp.Channel
	// consumeChans guarda cada *amqp.Channel de consumo aberto pela conexão
	// ATUAL (um por partição, ou Concurrency deles sobre a MESMA fila quando
	// KeyFunc == nil) — só para Close/reconnectLoop liberá-los; nenhuma
	// goroutine além da sua própria (consume) toca no seu respectivo canal
	// enquanto ele está vivo.
	consumeChans []*amqp.Channel

	// pubMu serializa a chamada de I/O de Publish em si (revisão da PR #23):
	// *amqp.Channel NÃO é seguro para publicação concorrente por múltiplas
	// goroutines (doc do driver amqp091-go) — Publish, ao contrário de
	// Subscribe (só popula um map protegido por mu, abaixo), pode ser
	// chamado concorrentemente por vários UseCases/Policies deste processo
	// publicando ao mesmo tempo no MESMO canal AMQP. Desde a task J3.3,
	// pubMu só cobre PublishWithContext — a leitura de qual *amqp.Channel
	// usar é connMu (acima), um mutex DIFERENTE de propósito (permite
	// Publishes concorrentes lerem r.ch em paralelo via RLock, serializando
	// só a chamada de rede em si).
	pubMu sync.Mutex

	// stopCh fecha quando Close() é chamado — supervise() e reconnectLoop()
	// o observam para parar de tentar reconectar/reobservar NotifyClose
	// (task J3.3): sem isso, Close() vazaria a goroutine de supervise para
	// sempre (ela ficaria bloqueada esperando o próximo NotifyClose de uma
	// conexão que ninguém mais vai fechar de propósito outra vez).
	// closeOnce garante que fechar stopCh duas vezes (Close chamado mais de
	// uma vez) nunca panica.
	stopCh    chan struct{}
	closeOnce sync.Once

	mu       sync.RWMutex
	handlers map[string][]func(ctx context.Context, ev runtime.Event) error
}

// NewRabbitMQChannel abre uma conexão AMQP para url, declara toda a
// topologia (exchange principal — fanout ou x-consistent-hash conforme
// cfg.KeyFunc — fila(s) principal(is), DLX de retry, retry queue e DLQ
// final, ver a doc do arquivo) e começa a consumir imediatamente (mesmo
// espírito de NewQueueChannel, que já sobe seus workers na construção).
// registry é repassado a decodeEnvelope para cada mensagem recebida (ver a
// doc do arquivo sobre R8/REQ-43.5 — o chamador monta o registry completo,
// esta função só o usa).
func NewRabbitMQChannel(url string, cfg RabbitMQConfig, registry map[string]EventFactory) (runtime.ChannelTransport, error) {
	r := &rabbitmqChannel{
		url:      url,
		cfg:      cfg,
		dlq:      dlqName(cfg.Exchange),
		registry: registry,
		handlers: make(map[string][]func(ctx context.Context, ev runtime.Event) error),
		stopCh:   make(chan struct{}),
	}

	conn, pubCh, consumeChans, err := r.dialAndSetup()
	if err != nil {
		return nil, fmt.Errorf("amqpruntime: NewRabbitMQChannel: %w", err)
	}

	r.connMu.Lock()
	r.conn, r.ch, r.consumeChans = conn, pubCh, consumeChans
	r.connMu.Unlock()

	// supervise (task J3.3, REQ-43.6, ver a doc do arquivo sobre
	// "Reconexão") roda pela vida inteira do canal, reconectando sozinho
	// quando a conexão cai inesperadamente.
	go r.supervise()

	return r, nil
}

// declareTopology declara, sobre conn, TODA a topologia deste canal
// (exchange principal — fanout ou x-consistent-hash conforme r.cfg.KeyFunc —
// fila(s) principal(is), DLX de retry, retry queue e DLQ final; ver a doc do
// arquivo) e devolve o nome de cada fila que precisa de um consumidor —
// função ÚNICA reusada tanto por NewRabbitMQChannel quanto por
// reconnectLoop (task J3.3): antes desta task esta lógica só existia inline
// dentro de NewRabbitMQChannel; extraída aqui para nunca duplicar entre os
// dois caminhos.
func (r *rabbitmqChannel) declareTopology(conn *amqp.Connection) ([]string, error) {
	cfg := r.cfg
	concurrency := int(effectiveConcurrency(cfg.Concurrency))

	setupCh, err := conn.Channel()
	if err != nil {
		return nil, fmt.Errorf("setup channel: %w", err)
	}
	defer setupCh.Close()

	retryExchange := retryExchangeName(cfg.Exchange)
	retryQueue := retryQueueName(cfg.Exchange)
	dlq := dlqName(cfg.Exchange)

	// DLX de retry + retry queue + DLQ final: sempre declaradas,
	// independente de orderBy (REQ-43.4 não é condicionado a ordenação).
	if err := setupCh.ExchangeDeclare(retryExchange, "fanout", true, false, false, false, nil); err != nil {
		return nil, fmt.Errorf("retry exchange declare: %w", err)
	}
	if _, err := setupCh.QueueDeclare(retryQueue, true, false, false, false, retryQueueArgs(cfg.Exchange, effectiveRetryTTL(cfg.RetryTTL))); err != nil {
		return nil, fmt.Errorf("retry queue declare: %w", err)
	}
	if err := setupCh.QueueBind(retryQueue, "", retryExchange, false, nil); err != nil {
		return nil, fmt.Errorf("retry queue bind: %w", err)
	}
	if _, err := setupCh.QueueDeclare(dlq, true, false, false, false, nil); err != nil {
		return nil, fmt.Errorf("dlq declare: %w", err)
	}

	var queueNames []string
	if cfg.KeyFunc != nil {
		if err := setupCh.ExchangeDeclare(cfg.Exchange, "x-consistent-hash", true, false, false, false, nil); err != nil {
			return nil, fmt.Errorf("exchange declare (consistent-hash): %w", err)
		}
		queueNames = partitionQueueNames(cfg.Queue, concurrency)
		for _, qn := range queueNames {
			if _, err := setupCh.QueueDeclare(qn, true, false, false, false, mainQueueArgs(retryExchange)); err != nil {
				return nil, fmt.Errorf("partition queue %q declare: %w", qn, err)
			}
			if err := setupCh.QueueBind(qn, consistentHashBindingKey(), cfg.Exchange, false, nil); err != nil {
				return nil, fmt.Errorf("partition queue %q bind: %w", qn, err)
			}
		}
	} else {
		if err := setupCh.ExchangeDeclare(cfg.Exchange, "fanout", true, false, false, false, nil); err != nil {
			return nil, fmt.Errorf("exchange declare (fanout): %w", err)
		}
		if _, err := setupCh.QueueDeclare(cfg.Queue, true, false, false, false, mainQueueArgs(retryExchange)); err != nil {
			return nil, fmt.Errorf("queue declare: %w", err)
		}
		if err := setupCh.QueueBind(cfg.Queue, "", cfg.Exchange, false, nil); err != nil {
			return nil, fmt.Errorf("queue bind: %w", err)
		}
		queueNames = make([]string, concurrency)
		for i := range queueNames {
			queueNames[i] = cfg.Queue
		}
	}
	return queueNames, nil
}

// dialAndSetup abre uma conexão NOVA a partir de r.url, declara a topologia
// inteira (declareTopology) e sobe um consumidor (goroutine + *amqp.Channel
// próprios, ver a doc do arquivo sobre por que cada consumidor tem o seu)
// por fila devolvida — usada tanto por NewRabbitMQChannel (construção)
// quanto por reconnectLoop (task J3.3): as DUAS só chamam isto, nenhuma
// reimplementa dial/declare/consume por conta própria. Numa falha em
// qualquer passo, fecha tudo que já tinha aberto até ali e devolve erro —
// nunca deixa a conexão/canais parcialmente montados vazando.
func (r *rabbitmqChannel) dialAndSetup() (conn *amqp.Connection, pubCh *amqp.Channel, consumeChans []*amqp.Channel, err error) {
	conn, err = amqp.Dial(r.url)
	if err != nil {
		return nil, nil, nil, fmt.Errorf("dial: %w", err)
	}
	fail := func(err error) (*amqp.Connection, *amqp.Channel, []*amqp.Channel, error) {
		conn.Close()
		return nil, nil, nil, err
	}

	queueNames, err := r.declareTopology(conn)
	if err != nil {
		return fail(err)
	}

	pubCh, err = conn.Channel()
	if err != nil {
		return fail(fmt.Errorf("publish channel: %w", err))
	}

	for _, qn := range queueNames {
		consumeCh, cErr := conn.Channel()
		if cErr != nil {
			pubCh.Close()
			for _, c := range consumeChans {
				c.Close()
			}
			return fail(fmt.Errorf("consume channel: %w", cErr))
		}
		if cErr = consumeCh.Qos(1, 0, false); cErr != nil {
			consumeCh.Close()
			pubCh.Close()
			for _, c := range consumeChans {
				c.Close()
			}
			return fail(fmt.Errorf("qos: %w", cErr))
		}
		msgs, cErr := consumeCh.Consume(qn, "", false, false, false, false, nil)
		if cErr != nil {
			consumeCh.Close()
			pubCh.Close()
			for _, c := range consumeChans {
				c.Close()
			}
			return fail(fmt.Errorf("consume %q: %w", qn, cErr))
		}
		consumeChans = append(consumeChans, consumeCh)
		go r.consume(msgs, consumeCh)
	}

	return conn, pubCh, consumeChans, nil
}

// supervise roda numa goroutine pela vida inteira do canal (task J3.3,
// REQ-43.6, ver a doc do arquivo sobre "Reconexão"): observa
// Connection.NotifyClose da conexão ATUAL (relida a cada volta do loop, já
// que reconnectLoop troca r.conn) e, ao detectar um fechamento que não veio
// de Close() (checagem de r.stopCh logo em seguida — Close() sempre fecha
// stopCh ANTES de fechar a conexão de verdade, então se stopCh já está
// fechado quando NotifyClose dispara, a causa foi Close(), não um blip de
// rede), aciona reconnectLoop.
func (r *rabbitmqChannel) supervise() {
	for {
		r.connMu.RLock()
		conn := r.conn
		r.connMu.RUnlock()

		// conn == nil (revisão da PR #25): reconnectLoop pode ter retornado
		// SEM reconectar (stopCh fechou durante a janela de reconexão,
		// Close() concorrente) — nesse caso r.conn continua nil e não há
		// nada pra supervisionar; Close() já está em andamento (é a única
		// forma de r.conn ficar nil depois de já ter conectado uma vez), então
		// retornar aqui é sempre correto, nunca perde um close legítimo.
		if conn == nil {
			return
		}

		closeNotify := conn.NotifyClose(make(chan *amqp.Error, 1))
		select {
		case <-r.stopCh:
			return
		case amqpErr := <-closeNotify:
			select {
			case <-r.stopCh:
				// Close() já estava em andamento — o fechamento é
				// deliberado, não um blip de rede; nada a reconectar.
				return
			default:
			}
			if amqpErr != nil {
				slog.Error("amqpruntime: conexão AMQP fechada inesperadamente, reconectando", "error", amqpErr)
			} else {
				slog.Error("amqpruntime: conexão AMQP fechada inesperadamente, reconectando")
			}
			r.reconnectLoop()
		}
	}
}

// reconnectLoop redial + re-declara a topologia inteira + reabre todos os
// consumidores (dialAndSetup, a MESMA função que NewRabbitMQChannel usa),
// com backoff exponencial (nextReconnectBackoff, mesmo idioma de
// DurableOutbox.Start) entre tentativas malsucedidas. Enquanto este loop
// roda, r.conn/r.ch/r.consumeChans ficam nil sob connMu — Publish trata isso
// como "sem conexão" e devolve erro em vez de bloquear ou desreferenciar nil
// (ver a doc do arquivo sobre "Reconexão"). Retorna quando reconecta com
// sucesso OU quando r.stopCh fecha (Close() chamado durante a janela de
// reconexão).
func (r *rabbitmqChannel) reconnectLoop() {
	r.connMu.Lock()
	oldConn, oldCh, oldChans := r.conn, r.ch, r.consumeChans
	r.conn, r.ch, r.consumeChans = nil, nil, nil
	r.connMu.Unlock()
	// A conexão antiga já está morta (é por isso que supervise nos trouxe
	// até aqui) — fecha o que sobrou best-effort, sem checar erro: o socket
	// subjacente já foi derrubado pelo broker/rede. oldConn TAMBÉM é fechada
	// explicitamente (revisão da PR #25) — mesmo já estando morta do lado do
	// broker, Close() ainda libera recursos do lado do CLIENTE (sockets TCP
	// residuais, goroutines internas do driver amqp091-go) que só um Close()
	// de verdade encerra.
	if oldCh != nil {
		_ = oldCh.Close()
	}
	for _, c := range oldChans {
		_ = c.Close()
	}
	if oldConn != nil {
		_ = oldConn.Close()
	}

	backoff := reconnectInitialBackoff
	for {
		select {
		case <-r.stopCh:
			return
		default:
		}

		conn, pubCh, consumeChans, err := r.dialAndSetup()
		if err != nil {
			slog.Error("amqpruntime: falha ao reconectar, tentando novamente", "error", err, "backoff", backoff)
			select {
			case <-r.stopCh:
				return
			case <-time.After(backoff):
			}
			backoff = nextReconnectBackoff(backoff)
			continue
		}

		r.connMu.Lock()
		select {
		case <-r.stopCh:
			// Close() rodou concorrentemente enquanto dialAndSetup() estava
			// em andamento (revisão da PR #25) — Close() já retornou sem
			// saber desta conexão nova, então ninguém mais vai fechá-la;
			// fecha AQUI, sob o lock, antes de devolvê-lo, pra nunca vazar o
			// socket/canais recém-abertos.
			r.connMu.Unlock()
			for _, c := range consumeChans {
				_ = c.Close()
			}
			_ = pubCh.Close()
			_ = conn.Close()
			return
		default:
		}
		r.conn, r.ch, r.consumeChans = conn, pubCh, consumeChans
		r.connMu.Unlock()
		slog.Info("amqpruntime: reconectado com sucesso")
		return
	}
}

// Subscribe implementa runtime.ChannelTransport — mesmo contrato de
// Dispatcher.Subscribe/queueChannel.Subscribe.
func (r *rabbitmqChannel) Subscribe(eventType string, handler func(ctx context.Context, ev runtime.Event) error) {
	r.mu.Lock()
	defer r.mu.Unlock()
	r.handlers[eventType] = append(r.handlers[eventType], handler)
}

// Publish implementa runtime.ChannelTransport: serializa ev no envelope de
// fio e publica na exchange — a routing key é KeyFunc(ev) quando o canal
// particiona por orderBy (ver a doc do arquivo), "" caso contrário
// (fanout — toda routing key entrega igual). Assíncrono do ponto de vista
// do broker (o mesmo "retorna assim que enfileirado" que
// QueueChannel.Publish já documenta), mas PublishWithContext só retorna
// depois que o broker confirma o recebimento no canal (não espera nenhum
// consumidor entregar).
//
// Durante uma reconexão (task J3.3, REQ-43.6) r.ch é nil — Publish devolve
// erro IMEDIATAMENTE nesse caso, nunca bloqueia nem desreferencia nil; ver a
// doc do arquivo sobre "Reconexão" para por que isso é suficiente (o relay
// do outbox, o único chamador que importa, já re-tenta).
func (r *rabbitmqChannel) Publish(ctx context.Context, ev runtime.Event) error {
	body, err := encodeEnvelope(ev)
	if err != nil {
		return err
	}
	routingKey := ""
	if r.cfg.KeyFunc != nil {
		routingKey = r.cfg.KeyFunc(ev)
	}

	// connMu (task J3.3): lê qual *amqp.Channel usar AGORA — RLock permite
	// Publishes concorrentes lerem em paralelo, só a troca (reconnectLoop)
	// precisa de exclusividade aqui.
	r.connMu.RLock()
	ch := r.ch
	exchange := r.cfg.Exchange
	r.connMu.RUnlock()
	if ch == nil {
		return fmt.Errorf("amqpruntime: Publish: sem conexão ativa (reconectando)")
	}

	// pubMu (revisão da PR #23): serializa o acesso ao *amqp.Channel — ver a
	// doc do campo em rabbitmqChannel.
	r.pubMu.Lock()
	defer r.pubMu.Unlock()
	return ch.PublishWithContext(ctx, exchange, routingKey, false, false, amqp.Publishing{
		ContentType: "application/json",
		Body:        body,
	})
}

// Close libera a conexão e todo *amqp.Channel abertos (canal de publish +
// um por consumidor) e para o supervisor de reconexão (task J3.3) — o
// wiring gerado (task J3.4/J6.2, "func run() error" com "defer Close()" por
// recurso, §design infra-providers "wiring multi-recurso fail-closed") vai
// chamar isto no unwind. Idempotente: chamar mais de uma vez nunca panica
// (closeOnce protege o fechamento de stopCh; conn/ch/consumeChans são lidos
// e zerados atomicamente sob connMu, então uma segunda chamada não encontra
// nada para fechar de novo).
func (r *rabbitmqChannel) Close() error {
	// Fecha stopCh ANTES de tocar a conexão — supervise()/reconnectLoop()
	// dependem dessa ordem para distinguir "Close() deliberado" de "blip de
	// rede" (ver a doc do arquivo sobre "Reconexão").
	r.closeOnce.Do(func() { close(r.stopCh) })

	r.connMu.Lock()
	conn, ch, chans := r.conn, r.ch, r.consumeChans
	r.conn, r.ch, r.consumeChans = nil, nil, nil
	r.connMu.Unlock()

	var errs []error
	if ch != nil {
		if err := ch.Close(); err != nil {
			errs = append(errs, err)
		}
	}
	for _, c := range chans {
		if err := c.Close(); err != nil {
			errs = append(errs, err)
		}
	}
	if conn != nil {
		if err := conn.Close(); err != nil {
			errs = append(errs, err)
		}
	}
	if len(errs) > 0 {
		return fmt.Errorf("amqpruntime: Close: %v", errs)
	}
	return nil
}

// consume roda pela vida inteira de UM *amqp.Channel de consumo (uma
// partição, ou um dos Concurrency consumidores concorrentes da fila única
// sem orderBy — ver a doc do arquivo), entregando cada mensagem ao(s)
// handler(s) assinados para seu eventType. Sucesso (decodificação E
// handler) ⇒ ack; falha ⇒ conta tentativas via x-death e decide entre
// nack(requeue=false) (mais uma volta pelo ciclo DLX+retry) ou publicar
// direto na DLQ final + ack (tentativas esgotadas) — nunca um
// nack(requeue=true) incondicional (poison pill clássico, ver a doc do
// arquivo sobre REQ-43.4).
func (r *rabbitmqChannel) consume(msgs <-chan amqp.Delivery, ch *amqp.Channel) {
	ctx := context.Background()
	for msg := range msgs {
		if r.deliver(ctx, msg) {
			_ = msg.Ack(false)
			continue
		}

		attempts := xDeathCount(msg.Headers) + 1
		if attempts >= effectiveMaxAttempts(r.cfg.MaxAttempts) {
			if pubErr := ch.PublishWithContext(ctx, "", r.dlq, false, false, amqp.Publishing{
				ContentType: "application/json",
				Body:        msg.Body,
			}); pubErr != nil {
				slog.Error("amqpruntime: falha ao publicar na DLQ final, mensagem re-enfileirada em vez de perdida", "error", pubErr, "attempts", attempts)
				_ = msg.Nack(false, false)
				continue
			}
			slog.Error("amqpruntime: tentativas esgotadas, mensagem movida para a DLQ final", "queue", r.dlq, "attempts", attempts)
			_ = msg.Ack(false)
			continue
		}
		_ = msg.Nack(false, false)
	}
}

// deliver decodifica msg e chama todo handler assinado para o eventType
// resultante — devolve true só quando AMBOS (decodificação e todo handler)
// sucederam.
func (r *rabbitmqChannel) deliver(ctx context.Context, msg amqp.Delivery) bool {
	ev, err := decodeEnvelope(msg.Body, r.registry)
	if err != nil {
		slog.Error("amqpruntime: mensagem não decodificou", "error", err)
		return false
	}

	r.mu.RLock()
	handlers := append([]func(ctx context.Context, ev runtime.Event) error(nil), r.handlers[ev.EventType()]...)
	r.mu.RUnlock()

	for _, h := range handlers {
		if err := h(ctx, ev); err != nil {
			slog.Error("amqpruntime: entrega falhou", "eventType", ev.EventType(), "error", err)
			return false
		}
	}
	return true
}
